# PRSG-002 — Design Concept

## Goals

Build the **navigation / traceability spine** for spec artifacts — a Maps-of-Content (MOC) layer — so that:

- When artifacts get collapsed (`.process/`) or relocated, **nothing becomes unreachable**: the MOC graph keeps every map linked to a parent.
- Decomposition stays **navigable** — you follow the map rather than memorizing the directory tree.
- Every **new** spec is born version-marked and parent-linked, so the enforcement lints apply to it from creation while pre-existing legacy specs are grandfathered.

Concretely, PRSG-002 ships:
1. Two MOC **template shapes** (roadmap-MOC, spec-MOC) carrying the frontmatter join-key contract.
2. A scaffold-time **skeleton writer**: `speckit-scaffold-spec` stamps a minimal `SPEC-MOC.md` on every new spec.
3. Two **version-gated Layer-1 lints**: orphan (a MOC lacking a valid `up:`) and stale-index (a MOC relative link that doesn't resolve).
4. **Namespace-aware ID normalization** that joins doc IDs to directory IDs across both the `SPEC-` and `PRSG-` conventions without collisions.

## Non-goals

- **Generating MOC content** — the down-link index, backlinks, and status integration are **PRSG-003**; the PRD-derived roadmap-MOC home note is **PRSG-004**. PRSG-002 produces *shapes and a skeleton*, not populated maps.
- **Retro-migration / backfill** of legacy specs and relocation of existing top-level exhaust into `.process/` — that is **PRSG-011** (the version-gating here is the deliberate upstream coupling that makes PRSG-011 safe).
- **Requiring `up:` on non-MOC docs** (`spec.md`, `plan.md`, `tasks.md`, `contracts/`) in v1 — those become reachable via the MOC down-index once PRSG-003 lands.
- **Wikilink support** — relative `[]()` links only; `[[wikilinks]]` in a MOC are a lint violation.
- Product code of any kind — this is a docs/process + plugin-skill change.

## Design Tree (Q&A log)

### Q1 — MOC minting rule (resolving the US1 conflict)
US1 said both "write the `SPEC-MOC.md` skeleton at creation (orphan prevention)" **and** "mint a spec-MOC only at the multi-slice squeeze point (single-slice → no MOC)" — a direct conflict, since version-gating reads `structureVersion` from `SPEC-MOC.md` and a spec with no marker is grandfathered/exempt.
- **Recommendation (chosen):** **Always write a minimal `SPEC-MOC.md`** at scaffold (`up:` + `structureVersion` + `spec_id`). Flesh it into a full navigation map only when the spec splits into multiple slices.
- **Why:** single-slice specs are the majority; if they carried no marker they'd all be grandfathered and the lints would never bite. The MOC doubles as the version-marker carrier, and a minimal file is cheap. "Single-slice → no MOC" is superseded for the *marker* file; it still applies to the richer navigation content.

### Q2 — structureVersion source-of-truth and value
- **Recommendation (chosen):** Plain **integer; v1 = 1**. Hardcode the literal in the lint script(s) with a "keep in sync" comment; `speckit-scaffold-spec` stamps the same `1`. Each lint fires for `structureVersion >= the version it shipped in`; an absent/no-marker spec is exempt.
- **Why:** trivial coupling at N=1; mirrors the repo's existing "copied verbatim, keep in sync" pattern (e.g., `extract_heading_section`). A shared version file or semver string is over-engineering for a value that changes ~once a year and only needs monotonic integer comparison.

### Q3 — ID normalization (PRSG- prefix + collision safety)
The scope text only says "strip `SPEC-`", but this repo uses `PRSG-` IDs (dirs like `prsg-002-moc-templates`) while legacy SPEC dirs carry no prefix (`013b-slug`). And `SPEC-002` + `PRSG-002` both exist, so blind prefix-stripping would collide on `002`.
- **Recommendation (chosen):** **Namespace-aware.** Normalize each ID/dir to `(namespace, number-suffix)`: lowercase, detect an optional leading alpha prefix (`spec-`/`prsg-`/…); a dir with **no** alpha prefix defaults to the legacy `spec` namespace. Match on **both** parts, with **exact-segment** compare on the number-suffix (`013a` ≠ `013a1`).
- **Why:** handles `PRSG-` correctly, prevents the `SPEC-002`/`PRSG-002` false match, and preserves the scope's exact-segment requirement that stops `SPEC-013A` matching the `013a1` dir.

### Q4 — Orphan lint scope in v1
- **Recommendation (chosen):** **MOC-only `up:` requirement.** The orphan lint requires a valid `up:` only on MOC files (`SPEC-MOC.md` / spec-MOCs) within version-gated specs. Non-MOC docs are not required to carry `up:` in v1; `.process/**` exhaust is exempt.
- **Why:** `spec.md`/`plan.md`/`tasks.md` are generated by SpecKit without `up:`; requiring it on them would couple to SpecKit templates and force frontmatter injection. The MOC down-index that makes other docs reachable is PRSG-003. The MOC-only check keeps the navigation spine connected upward without that coupling.

### Q5 — PRSG-002 ↔ PRSG-003/004 boundary, and the `up:` target
`SPEC-MOC.md`'s `up:` must resolve to a real file, but the roadmap-MOC home note is PRSG-004's deliverable.
- **Recommendation (chosen):** PRSG-002 ships **only** the template shapes, the scaffold-time skeleton writer, the two lints, ID normalization, and version-gating — **no generated MOC content**. `SPEC-MOC.md`'s `up:` points at the **existing** `*-technical-roadmap.md` so every link resolves from day one; PRSG-004 repoints it to the roadmap-MOC home note when that lands.
- **Why:** keeps PRSG-002 tight and the navigation graph valid immediately, with no dangling links to trip the stale-index lint, and no overlap with PRSG-003 (index/backlinks/status) or PRSG-004 (PRD-derived home note).

### Q6 — Where the lints run and what they scan
- **Recommendation (chosen):** Ship the lints as **Layer-1 scripts wired into `tests/run-all.sh`**, scanning **this repo's real spec trees** (`docs/ai/specs/` + `specs/`), with committed fixtures for the logic tests. Version-gating keeps the legacy specs green on day one; the same scripts are runnable in any consuming project's CI. **Hard-fail** (exit nonzero) on violation in a version-gated spec; no-marker specs are silently skipped.
- **Why:** the "first upgrade red-fails CI on legacy specs" rationale only makes sense if the lints scan real specs — so this repo dogfoods the contract, and the lints are reusable downstream. Matches how the plugin already dogfoods `.process/` `.gitattributes`.

### Q7 — MOC template location and consumption
- **Recommendation (chosen):** Add `roadmap-moc-template.md` + `spec-moc-template.md` to the **plugin templates dir** next to the existing `workflow-template.md`; `speckit-scaffold-spec` reads the spec-MOC template and writes the minimal `SPEC-MOC.md` via token substitution — the exact mechanism it already uses for `workflow-template.md`. No new preset.
- **Why:** matches the existing pattern; MOC is not a SpecKit-core template type, so a `specify preset resolve` path or project-local `.specify/templates/` copy would add plumbing and drift for no gain.

### Q8 — Frontmatter contract: enforced vs carried
Contract fields: `up` / `related` / `status` / `rank` / `spec_id` / `structureVersion`.
- **Recommendation (chosen):** v1 lints enforce only the **3 load-bearing fields**: `up` (valid relative path — the orphan check), `structureVersion` (integer; its presence is the version gate), and `spec_id` (must match the dir via namespace-aware normalization — validates the join). `status`, `rank`, `related` ship in the template but are optional/unenforced in v1; PRSG-003/004 populate and consume them.
- **Why:** minimal lint surface that exactly matches the MOC-only orphan + version-gate decisions, while the template still carries the full contract for downstream specs. Notably, enforcing `spec_id` is what actually validates the headline ID-normalization feature.

### Recorded defaults (minor items, agreed at wrap-up)
- **(a) Wikilink rejection:** the stale-index lint flags any `[[wikilink]]` inside a MOC as a violation — the contract is relative `[]()` links only.
- **(b) Codex parity:** mirror the `speckit-scaffold-spec` skeleton-writing change in `codex-skills/speckit-scaffold-spec/SKILL.md`. The MOC templates and the lint shell scripts are shared / runtime-agnostic (one copy each).
- **(c) Stale-index resolution:** the stale-index lint checks that every relative link target in a MOC resolves to an existing file.
