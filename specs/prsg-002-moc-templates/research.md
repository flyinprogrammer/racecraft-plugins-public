## D1 â€” MOC minting rule: always write a minimal marker

- **Decision**: Always write a minimal `SPEC-MOC.md` at scaffold time (`up:` + `structureVersion: 1` + `spec_id`). Flesh it into a richer navigation map only when a spec later splits into multiple slices (deferred).
- **Rationale**: The marker doubles as the version-gate carrier. Single-slice specs are the majority; if they carried no marker they would all be grandfathered and the lints would never bite. A minimal file is cheap. (Resolves the US1 internal conflict between "write at creation for orphan prevention" and "mint only at the multi-slice squeeze point".)
- **Alternatives considered**: "Single-slice â†’ no MOC file" â€” rejected for the *marker* file because it defeats version-gating; it still applies to the richer navigation *content* (deferred to decomposition specs).

## D2 â€” structureVersion source-of-truth and value

- **Decision**: Plain integer; v1 = `1`. Hardcode the literal in the lint script(s) with a "keep in sync" comment; the scaffold skill stamps the same `1` with its own "keep in sync" comment. Each lint fires for `structureVersion >= the version it shipped in`; an absent/no-marker spec is exempt. No shared version file, no semver string.
- **Rationale**: Trivial coupling at N=1; mirrors the repo's existing "copied verbatim, keep in sync" pattern (e.g., `extract_heading_section`, and the gate-glob â†” `.gitattributes` duplication guarded by `validate-process-gitattributes.sh`). Monotonic integer comparison is all that is needed.
- **Alternatives considered**: A shared version file or semver string â€” rejected as over-engineering for a value that changes ~once a year (YAGNI / KISS, constitution VI).

## D3 â€” Namespace-aware ID normalization (collision safety)

- **Decision**: Normalize each doc ID / directory name to a `(namespace, number-suffix)` pair: lowercase; split on `-`; if the first dash-delimited segment is all-alpha it is the namespace and the next segment is the number-suffix, otherwise namespace = `spec` and the first segment is the number-suffix. Compare the number-suffix as an **opaque whole segment** (byte-equality of the full segment â€” no `[0-9]+[a-z]*` sub-parse). A match requires BOTH parts to agree.
- **Rationale**: This repo uses `PRSG-` IDs (dirs like `prsg-002-moc-templates`) alongside legacy unprefixed dirs (`013b-slug`), and `SPEC-002` + `PRSG-002` both exist. Blind prefix-stripping would collide on `002`. Namespace-awareness keeps `(prsg, 002)` distinct from `(spec, 002)`; opaque-segment compare preserves the exact-segment requirement so `013a` != `013a1`.
- **Canonical examples**: `006a-uat-skeleton` â†’ `(spec, 006a)`; `prsg-002-moc-templates` â†’ `(prsg, 002)`; bare id `PRSG-002` â†’ `(prsg, 002)` (so PRSG-002's marker `spec_id` matches its directory); `002-pr-checks-workflow` â†’ `(spec, 002)` (does NOT match `PRSG-002`).
- **Alternatives considered**: Strip `SPEC-` only (the original scope wording) â€” rejected: ignores `PRSG-` and false-matches `SPEC-002`/`PRSG-002`. Sub-parsing digits from trailing letters â€” rejected: would truncate `013a1`â†’`013a` and create a false match.

## D4 â€” Orphan lint scope in v1: MOC files only

- **Decision**: The orphan lint requires a valid `up:` only on MOC files (exact filename `SPEC-MOC.md` in v1) inside version-gated specs. Non-MOC docs (`spec.md`, `plan.md`, `tasks.md`, `contracts/**`) are NOT required to carry `up:` in v1; `.process/**` exhaust is exempt.
- **Rationale**: `spec.md`/`plan.md`/`tasks.md` are generated by SpecKit without `up:`; requiring it on them would couple to SpecKit templates and force frontmatter injection. The MOC down-index that makes other docs reachable is PRSG-003. The MOC-only check keeps the spine connected upward without that coupling.
- **Alternatives considered**: Require `up:` on all docs â€” rejected (couples to SpecKit templates; out of v1 scope).

## D5 â€” PRSG-002 â†” PRSG-003/004 boundary, and the `up:` target

- **Decision**: PRSG-002 ships only the template shapes, the scaffold-time skeleton writer, the two lints, ID normalization, and version-gating â€” no generated MOC content. The marker's `up:` points at the **existing** `*-technical-roadmap.md` so every link resolves from day one. PRSG-004 later repoints it to the roadmap-MOC home note.
- **Rationale**: Keeps PRSG-002 tight and the navigation graph valid immediately, with no dangling link to trip the stale-index lint, and no overlap with PRSG-003 (index/backlinks/status) or PRSG-004 (PRD-derived home note).
- **Concrete target for this spec's own marker**: `../../docs/ai/specs/pr-size-governance-technical-roadmap.md` (verified reachable from `specs/prsg-002-moc-templates/`).
- **Alternatives considered**: Point `up:` at a roadmap-MOC home note now â€” rejected: that note is PRSG-004's deliverable and does not exist yet (would dangle).

## D6 â€” Where the lints run and what they scan

- **Decision**: Ship the lints as Layer-1 scripts wired into `tests/run-all.sh`, scanning this repo's real spec trees (`docs/ai/specs/` + `specs/`), with committed fixtures for the logic. Hard-fail (exit nonzero) on a violation in a version-gated spec; no-marker specs are silently skipped. The same scripts are runnable in any consuming project's checks.
- **Rationale**: The "first upgrade red-fails CI on legacy specs" concern only makes sense if the lints scan real specs â€” so this repo dogfoods the contract. Matches how the plugin already dogfoods `.process/` `.gitattributes` via `validate-process-gitattributes.sh`.
- **Repo-root resolution**: mirror the existing idiom `REPO_ROOT="$(cd "$(dirname "$0")/../../.." && pwd)"` (layer1-structural â†’ tests â†’ speckit-pro â†’ repo root), since `run-all.sh` runs from `speckit-pro/` but the spec trees live at repo root.
- **Alternatives considered**: Fixtures only (no real-tree scan) â€” rejected: would not validate the version-gating safety property against actual legacy specs.

## D7 â€” MOC template location and consumption

- **Decision**: Add `roadmap-moc-template.md` + `spec-moc-template.md` to the plugin templates dir `speckit-pro/skills/speckit-coach/templates/` (next to `workflow-template.md`). `speckit-scaffold-spec` reads the spec-MOC template and writes the minimal `SPEC-MOC.md` via token substitution â€” the exact mechanism it already uses for `workflow-template.md`. No new preset.
- **Rationale**: Matches the existing pattern. MOC is not a SpecKit-core template type, so a `specify preset resolve` path or a project-local `.specify/templates/` copy would add plumbing and drift for no gain (KISS).
- **Alternatives considered**: A new SpecKit preset, or a project-local template copy â€” both rejected as unnecessary plumbing.

## D8 â€” Frontmatter contract: enforced vs carried

- **Decision**: v1 lints enforce only the 3 load-bearing fields: `up` (present + well-formed relative link â†’ orphan check; resolution â†’ stale-index check), `structureVersion` (integer; presence/value is the version gate), and `spec_id` (must namespace-match its directory â†’ validates the join). `status`, `rank`, `related` ship in the templates but are optional/unenforced in v1.
- **Rationale**: Minimal lint surface that exactly matches the MOC-only orphan + version-gate decisions, while the template still carries the full contract for downstream specs (PRSG-003/004 populate and consume `status`/`rank`/`related`). Enforcing `spec_id` is what actually validates the headline ID-normalization feature.
- **Alternatives considered**: Enforce all six fields in v1 â€” rejected (premature; YAGNI).

## D9 (recorded defaults)

- **(a) Wikilink rejection**: the stale-index lint flags any `[[wikilink]]` inside a MOC as a violation in its own right â€” the contract is relative `[]()` links only.
- **(b) Codex parity**: mirror the `speckit-scaffold-spec` skeleton-writing change in `codex-skills/speckit-scaffold-spec/SKILL.md`. The MOC templates and the lint shell scripts are shared / runtime-agnostic (one copy each).
- **(c) Stale-index resolution scope**: the stale-index lint resolves every relative `[]()` target in a MOC â€” **including the frontmatter `up:` value** â€” so a dangling `up:` is itself a stale-index violation. This is the division of labor with the orphan lint: orphan owns *presence + well-formedness* of `up:`; stale-index owns *resolution* of all relative targets.

## D10 â€” `spec_id`â†”directory naming (maintainer decision, 2026-06-06)

- **Decision**: `SPEC-MOC.md`'s `spec_id` carries the **roadmap identity** (`PRSG-002`), and the contract directory is **branch-named** to namespace-match it (`specs/prsg-002-moc-templates/`), NOT a sequential number. The scaffold creates `specs/<branch-name>/` from the branch name, not from `create-new-feature.sh`'s sequential numbering.
- **Rationale**: Under FR-018, `PRSG-002` â†’ `(prsg, 002)` and `prsg-002-moc-templates` â†’ `(prsg, 002)` **match**, so the dogfooded lints stay green on this spec with no special-casing. `create-new-feature.sh`'s default numbering would produce `(spec, NNN)` and break the join. Safe because autopilot's Specify phase skips `create-new-feature.sh` when already on a feature branch, so there is no collision with a later auto-numbered directory. Pre-existing legacy numeric dirs carry no `SPEC-MOC.md` and stay grandfathered/exempt.
- **Alternatives considered**: A roadmap-idâ†”directory mapping layer â€” rejected: unnecessary; the namespace-aware normalizer already makes branch-named dirs match their `spec_id`.
