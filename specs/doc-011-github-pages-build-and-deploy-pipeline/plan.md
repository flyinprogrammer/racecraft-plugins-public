# Implementation Plan: GitHub Pages Build-And-Deploy Pipeline

**Branch**: `doc-011-github-pages-build-and-deploy-pipeline` | **Date**: 2026-06-23 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `specs/doc-011-github-pages-build-and-deploy-pipeline/spec.md`

## Summary

DOC-011 adds one deploy-ready staging capability for the existing Astro/Starlight docs site: a standard GitHub Pages Actions workflow, docs validation before artifact upload, staging indexing protection, a CI/CD verification runbook, and concise agent guidance. The plan keeps Pages repository settings as manual operator setup, preserves the current staging `site`/`base` assumptions until DOC-012, and uses broad explicit workflow paths to reduce missed docs-impacting deploys while excluding fixture-heavy churn.

## Technical Context

**Language/Version**: Docs-site JavaScript ESM on Node >=22.12; GitHub Actions YAML; Markdown operator guidance

**Primary Dependencies**: Astro 6.4.6, Starlight 0.40.0, `@astrojs/check`, `starlight-links-validator`, Playwright 1.61.0, pnpm 10.25.0 via Corepack, standard GitHub Pages Actions

**Storage**: Checked-in repository files only; GitHub Pages stores the uploaded `docs-site/dist` static artifact outside repository source control

**Testing**: `pnpm --dir docs-site validate` after `pnpm --dir docs-site install --frozen-lockfile` and Chromium-only Playwright install; YAML/file-contract review for workflow paths, permissions, artifact path, and Pages environment

**Target Platform**: GitHub Actions `ubuntu-latest` with Node 22 and GitHub Pages staging environment

**Project Type**: Documentation/static-site CI/CD workflow and operator runbook

**Performance Goals**: No new runtime code path; deployment publishes the already-built static output after validation without adding redundant build scripts

**Constraints**: Use standard Pages Actions; do not automate repository Pages settings; do not introduce repository secrets, deploy keys, personal access tokens, GitHub App tokens, custom token inputs, or broad/default write permissions; preserve staging URL and noindex guard until DOC-012; install only Chromium browser dependencies needed by docs smoke tests; keep build/upload and deploy as separate jobs; clean `docs-site/dist` before validation and upload only the artifact generated by the same successful workflow run; use one fixed staging concurrency group with `cancel-in-progress: true` for `main` deploys while isolating skipped non-`main` manual dispatch runs from that group

**Scale/Scope**: One docs-site deploy workflow for one marketplace documentation site; explicit broad paths cover docs-site, generated-reference inputs, marketplace/plugin source surfaces, root scripts, generated payloads, and workflow files with fixture-heavy exclusions

**Reviewability Budget**: Primary surface: docs/process; secondary surface: infra; projected reviewable LOC: 180-320; projected production files: 5; projected total files: 8; setup gate warning recorded because the slice crosses docs/process plus infra surfaces, but it remains one deploy-ready capability with one observable staging outcome

## Phase 0 Research Summary

See [research.md](./research.md). All planning unknowns are resolved:

- Standard GitHub Pages Actions are sufficient.
- `pnpm --dir docs-site validate` remains the deploy gate.
- Node 22 plus Corepack pnpm 10.25.0 matches the docs-site package manager.
- Chromium-only Playwright install is the minimum CI setup needed for smoke tests.
- `docs-site/dist` is the artifact path because Astro uses the default output directory.
- Broad explicit `push.paths` are chosen to avoid missed generated-reference deploys, with fixture-heavy exclusions.
- `robots.txt` is only policy/signaling for a Pages project path; the Starlight noindex meta tag is the primary indexing guard.

## Phase 1 Design Summary

See [data-model.md](./data-model.md), [contracts/deploy-docs-workflow.md](./contracts/deploy-docs-workflow.md), and [quickstart.md](./quickstart.md).

Post-design constitution check remains PASS with the same reviewability WARN:

- No new custom deploy script or dependency is introduced.
- The planned workflow uses least-privilege `contents: read`, `pages: write`, and `id-token: write`.
- The deploy path uses no configured repository or organization secrets, deploy keys, personal access tokens, GitHub App tokens, custom `token:` inputs, or broad/default write permissions.
- Build/upload and deploy are separate jobs, with deploy depending on the validated artifact job.
- Push and `workflow_dispatch` runs share the same clean validation and artifact path; the deploy job consumes only the same-run Pages artifact and never rebuilds or uploads.
- Overlapping staging deploy runs use a fixed `github-pages` staging concurrency group with newer runs canceling older in-progress or pending runs.
- Manual Pages setup remains operator documentation rather than a CLI/API mutation.
- DOC-012 remains the only spec allowed to remove staging indexing protection or move to the launch domain.
