### User Story 1 - speckit-prd emits a roadmap-MOC home note (Priority: P1)

This story adds a third output artifact to the `speckit-prd` skill. When a developer asks the plugin to write a new PRD and technical roadmap, it now also writes a "roadmap-MOC home note" — a single navigable map for the whole spec tree — alongside the other two documents.

2. In the same file, find step 5 ("Emit the roadmap-MOC home note"). Confirm it describes writing the home note with two zones: a curated epics zone (auto-scaffolded from the roadmap's phases, zero new interview questions) and a sentinel-bounded GENERATED INDEX zone.

3. Confirm the skill instructs adding a reciprocal link from the technical-roadmap back to the home note, so both documents link to each other.

4. Confirm the skill says: when the auto-derived scaffold yields more than roughly 10 epics, print a one-line advisory and still write the file — the cap is never a block.

5. Confirm the skill says: when the roadmap has no phase grouping (a flat catalog), emit a single "Specs" epic containing all specs plus an advisory note to group them.

6. Confirm the skill says: do NOT emit a home note when running against an existing/legacy roadmap. The home note is for freshly authored roadmaps only.

7. Open `speckit-pro/codex-skills/speckit-prd/SKILL.md` and confirm it describes the same three-artifact output and the same two-zone home-note emit step. The wording may differ; the behavior described must be the same.

## Negative-Path Tests

Try each of these by inspection or by running the Layer 4 fixture and reading the test descriptions:

- **Roadmap with no phase grouping (flat catalog):** the curated scaffold should collapse to a single "Specs" epic with an advisory note suggesting the author group the specs. Look for this case in `speckit-prd/SKILL.md` (US1 step 5).

- **More than roughly 10 epics:** `speckit-prd` should print a one-line advisory and still write the home note — the epic count never blocks the write. Look for this behavior in `speckit-prd/SKILL.md` (US1 step 4).

- **Spec with blank or missing `status`:** the INDEX row is still emitted. Only the status field is blank; the row has a link and the middle-dot separator. The row is NOT dropped. Confirmed by the Layer 4 fixture (US3 step 3).

- **Spec with blank or missing `spec_id`:** the INDEX row is SKIPPED entirely. An absent `spec_id` means there is no usable link text and no sort key — fabricating a zero-width row would be misleading. The existing structural linter flags this as a defect in the spec itself; the INDEX generator simply skips it. Confirmed by inspecting the generator logic (US3 step 7).

- **Legacy/non-version-marked SPEC-MOC:** skipped by the INDEX. The generator requires the version marker (`structureVersion`) to be present; any spec without it is silently ignored, not an error. Confirmed by the Layer 4 fixture (US3 step 4).

- **Empty or absent `specs/` directory:** the generator runs cleanly with no INDEX rows and no error. Consistent with the existing behavior before this change.

- **Gated home note missing its INDEX sentinel pair:** the generator exits with code 2 and writes nothing. This is a safety guard — without the INDEX sentinels the generator would inject the wrong zones. Confirmed by inspecting the generator (US3 step 9).

- **Running the generator against this repository:** there is no `docs/ai/specs/*-roadmap-MOC.md` file in this repository yet (backfill onto the existing roadmap is owned by a later spec). Running the generator here is a clean no-op for the home-note path — it will update the per-spec SPEC-MOC indexes as before, and silently skip the home-note pass because the glob finds no matches.
