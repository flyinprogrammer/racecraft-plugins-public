### User Story 1 - speckit-prd emits a roadmap-MOC home note [US1] (Priority: P1)

When `speckit-prd` authors a new PRD and technical-roadmap, it ALSO writes a roadmap-MOC
"home note" at `docs/ai/specs/<slug>-roadmap-MOC.md` — a single navigable map for the
whole spec tree. The home note carries two zones: (a) a **human-curated epics zone**,
auto-derived as an editable scaffold from the roadmap's existing phase/tier grouping with
a one-line advisory "Why" per epic; and (b) a **sentinel-bounded GENERATED INDEX zone**.
`speckit-prd` also adds one reciprocal link from the technical-roadmap back to the home
note, so the two top-level documents are mutually reachable. The home note is emitted only
when `speckit-prd` authors a fresh PRD + roadmap (new-roadmaps-only).

**Why this priority**: This is the artifact PRSG-004 exists to deliver — the one map that
makes decomposition add traceability instead of cognitive load. Without it, US2 has nothing
to teach about and US3 has no consumer for its activated INDEX renderer.

**Independent Test**: Run `speckit-prd` against a fresh fixture roadmap and confirm a
third file `docs/ai/specs/<slug>-roadmap-MOC.md` is written alongside the PRD and
technical-roadmap, containing both an editable curated epics zone seeded from the roadmap's
phases and an empty-but-sentinel-bounded GENERATED INDEX zone, plus a reciprocal link added
to the technical-roadmap. Delivers value as a navigable home note even before US3 fills the
INDEX.

**Acceptance Scenarios**:

1. **Given** a fresh roadmap whose specs are grouped into phases/tiers, **When**
   `speckit-prd` finishes authoring the PRD + technical-roadmap, **Then** it writes
   `docs/ai/specs/<slug>-roadmap-MOC.md` containing a curated epics zone scaffolded from
   those phase groupings (one epic per phase, the phase's spec links, and an advisory "Why"
   placeholder per epic) and a sentinel-bounded GENERATED INDEX zone.
2. **Given** the roadmap has no phase grouping (a flat catalog), **When** `speckit-prd`
   derives the curated zone, **Then** it emits a single "Specs" epic containing all specs
   plus an advisory note suggesting the author group them.
3. **Given** the auto-derived scaffold yields more than ~10 epics, **When** `speckit-prd`
   emits the home note, **Then** it prints a one-line advisory (e.g. "11 epics — consider
   consolidating; >~10 strains navigability") but still writes the file.
4. **Given** `speckit-prd` has written the home note, **When** the emission completes,
   **Then** the home note's frontmatter `up:` points at the technical-roadmap via a
   relative `[]()` link, and the technical-roadmap carries a one-line reciprocal link to
   the home note.
5. **Given** `speckit-prd` runs against an existing/legacy roadmap (not a freshly authored
   one), **When** it completes, **Then** it does NOT backfill a home note for that legacy
   roadmap (backfill is owned by a later spec).

### User Story 2 - speckit-coach teaches the two-zone structure [US2] (Priority: P3) [P]

`speckit-coach` explains the home note's two-zone model: the **curated** epics zone is
hand-authored and editable, while the **generated** INDEX zone is machine-regenerated and
must never be hand-edited. Coaching also conveys the "cap epics below ~10" guardrail as an
advisory navigability guideline (warn, not block).

**Why this priority**: Pure documentation/coaching prose with no runtime behavior; it
depends on US1's artifact existing to be meaningful but ships independently and can be
authored in parallel with the code work.

**Independent Test**: Ask `speckit-coach` about the roadmap-MOC home note and confirm it
distinguishes the curated zone (hand-edited) from the generated zone (never hand-edited,
regenerated by the generator) and states the advisory ~10-epic cap.

**Acceptance Scenarios**:

1. **Given** a developer asks `speckit-coach` how the roadmap-MOC home note is structured,
   **When** it answers, **Then** it describes the curated-vs-generated two-zone split and
   states that the generated INDEX zone is regenerated and never hand-edited.
2. **Given** a developer asks about epic count, **When** `speckit-coach` answers, **Then**
   it conveys the advisory "cap epics below ~10" guideline and that it is a warning, not a
   block.

### User Story 3 - Activate render_index() for a deterministic GENERATED INDEX [US3] (Priority: P2)

Activate the deliberately dormant `render_index()` in `generate-spec-index.sh` and teach
its `main()` to also process the roadmap-MOC home note, so the home note's GENERATED INDEX
zone is produced deterministically: one row per in-scope spec, `link + status`, in
normalized-ID ascending order. The existing per-spec-MOC code path MUST remain
byte-identical so the pinned PRSG-003 contracts stay green.

**Why this priority**: This is the enabling change that makes US1's INDEX zone non-empty
and trustworthy. US1 can ship its curated zone and an empty sentinel-bounded INDEX without
it, but the home note's at-a-glance index value only arrives once render_index is live.
US1 and US3 are ordered: US3 fills what US1 frames.

**Independent Test**: Given a fixture repo with several version-marked SPEC-MOCs and a
home note carrying the INDEX sentinels, run the generator and confirm the INDEX zone is
filled with one `- [<spec_id>](rel/SPEC-MOC.md) · <status>` row per spec in normalized-ID
ascending order; run it a second time and confirm a zero-byte diff; confirm every existing
spec-MOC file is byte-identical to before.

**Acceptance Scenarios**:

1. **Given** a home note with the INDEX sentinels and a set of version-marked SPEC-MOCs,
   **When** the generator runs, **Then** the INDEX zone contains exactly one row per
   in-scope spec, each row `- [<spec_id>](rel/SPEC-MOC.md) · <status>` with `spec_id` and
   `status` read from that SPEC-MOC's frontmatter, ordered normalized-ID ascending.
2. **Given** the generator has just regenerated the home note, **When** it runs again with
   no intervening edits, **Then** the INDEX zone is byte-identical (a second run is a
   zero-byte diff).
3. **Given** the same committed SPEC-MOCs are present, **When** the generator runs,
   **Then** every per-spec SPEC-MOC.md file is byte-identical to its pre-activation output
   (the spec-MOC code path is unchanged).
4. **Given** a SPEC-MOC whose `status` frontmatter field is empty or missing, **When** the
   INDEX row is rendered, **Then** the row is still emitted with its link and an
   empty/blank status (no crash, no row dropped).
5. **Given** a spec directory whose SPEC-MOC is legacy/non-version-marked, **When** the
   generator runs, **Then** that spec is skipped (not indexed), consistent with the
   existing gating.
6. **Given** the generator runs and the relative link from the home note to a SPEC-MOC is
   constructed, **When** the row is rendered, **Then** the link is a relative `[]()` target
   resolving `../../../specs/<dir>/SPEC-MOC.md` — never a `[[wikilink]]`.

### Edge Cases

- **No phase grouping in the roadmap (flat catalog)**: the curated scaffold collapses to a
  single "Specs" epic listing all specs, with an advisory note to group them.
- **Epic count exceeds ~10**: advisory warning printed at emit time; the file is still
  written (the cap is never a block).
- **SPEC-MOC with empty or missing `status`**: the INDEX row renders with link + blank
  status; the row is not dropped.
- **Legacy/non-version-marked SPEC-MOC**: skipped by the INDEX (consistent with the
  existing version-gating); not an error.
- **Empty or absent `specs/` tree**: the generator is a clean no-op (no INDEX rows, no
  error), consistent with the existing driver.
- **Existing/legacy roadmap encountered by speckit-prd**: no home-note backfill (deferred
  to the later retro-migration spec).

## Requirements *(mandatory)*

### Functional Requirements

**Emission — speckit-prd (US1)**

- **FR-001 [US1]**: `speckit-prd` MUST, when it authors a new PRD and technical-roadmap,
  also write a roadmap-MOC home note at `docs/ai/specs/<slug>-roadmap-MOC.md` as a
  separate file (NOT zones injected into the prose technical-roadmap).
- **FR-002 [US1]**: The home note MUST contain two zones: a human-curated epics zone and a
  sentinel-bounded GENERATED INDEX zone, using the same sentinel framing the generator
  recognizes. `speckit-prd` MUST emit **only the INDEX sentinel pair** (not the PRS or
  BACKLINKS pairs): the generator's inject-if-missing path fires only when all three
  GENERATED pairs are absent, so a home note carrying just the INDEX pair takes the
  whole-zone rewrite path and fills only the INDEX.
- **FR-003 [US1]**: `speckit-prd` MUST auto-derive the curated epics zone from the
  roadmap's existing phase/tier grouping — one epic per phase, the phase's spec links, and
  a one-line advisory "Why" placeholder per epic — as an editable scaffold, adding ZERO new
  interview questions.
- **FR-004 [US1]**: When the roadmap has no phase grouping, `speckit-prd` MUST emit a
  single "Specs" epic containing all specs plus an advisory note suggesting the author
  group them.
- **FR-005 [US1]**: When the auto-derived scaffold yields more than approximately 10 epics,
  `speckit-prd` MUST print a one-line advisory and MUST still write the file (advisory only
  — never a block, never a CI lint).
- **FR-006 [US1]**: The home note's frontmatter `up:` MUST point at the technical-roadmap
  (`<slug>-technical-roadmap.md`) via a relative `[]()` link, and `speckit-prd` MUST add a
  one-line reciprocal link from the technical-roadmap back to the home note, so the two
  top-level documents are mutually reachable.
- **FR-007 [US1]**: `speckit-prd` MUST emit the home note only for freshly authored
  roadmaps; it MUST NOT backfill a home note onto an existing/legacy roadmap.
- **FR-008 [US1]**: `speckit-prd` MUST NOT change each spec-MOC's `up:` value, the spec-MOC
  template, or scaffold-spec; the home note is a downward index + curated layer that
  cross-links with the roadmap, leaving the existing upward navigation contract untouched.

**Coaching — speckit-coach (US2)**

- **FR-009 [US2]**: `speckit-coach` MUST teach the home note's curated-vs-generated
  two-zone split, including that the curated epics zone is hand-authored/editable and the
  GENERATED INDEX zone is machine-regenerated and never hand-edited.
- **FR-010 [US2]**: `speckit-coach` MUST teach the "cap epics below ~10" guardrail as an
  advisory navigability guideline (warn, not block).

**Generated INDEX — render_index activation (US3)**

- **FR-011 [US3]**: The generator MUST activate `render_index()` so the home note's
  GENERATED INDEX zone is produced from committed files only (deterministic; no network,
  no `gh`). Activation MUST be **context-scoped**: `render_index` is already invoked today
  against every spec-MOC's present-but-empty INDEX zone (where it returns empty), so it MUST
  emit repo-wide rows **only when the target is the roadmap-MOC home note** and continue to
  render empty for spec-MOCs (see FR-018).
- **FR-012 [US3]**: Each INDEX row MUST be exactly `- [<spec_id>](rel/SPEC-MOC.md) ·
  <status>`, where `spec_id` and `status` are read from each SPEC-MOC's frontmatter via the
  existing frontmatter accessor, and the separator is the U+00B7 MIDDLE DOT (matching the
  existing renderer).
- **FR-013 [US3]**: INDEX rows MUST be ordered normalized-ID ascending, using the existing
  ID-normalization helper.
- **FR-014 [US3]**: Each INDEX link MUST be a relative `[]()` target resolving
  `../../../specs/<dir>/SPEC-MOC.md` (home note in `docs/ai/specs/`, spec tree in repo-root
  `specs/`) — never a `[[wikilink]]`.
- **FR-015 [US3]**: A SPEC-MOC with an empty or missing `status` MUST still produce an
  INDEX row (link + blank status); the row MUST NOT be dropped. The `status` field is
  display-only (it is neither the link text nor the sort key), so a blank value degrades the
  row gracefully rather than invalidating it.
- **FR-015a [US3]**: `spec_id` is NOT symmetric with `status`. A non-empty `spec_id` is a
  PRSG-002 gating invariant: an absent or empty `spec_id` in a version-gated marker is itself
  a PRSG-002 lint violation (`specs/prsg-002-moc-templates/contracts/id-normalization-grammar.md`
  — "a marker with no join key cannot satisfy the join"), and `spec_id` is BOTH the INDEX
  row's visible link text AND its `moc_normalize` sort key. Therefore a gated SPEC-MOC whose
  `spec_id` is absent or empty MUST be **skipped from the INDEX** (no row emitted) — distinct
  from the empty-`status` case, which still emits a row. PRSG-004 adds no new validation for
  this: the existing PRSG-002 `spec_id` lint remains the authority that flags the offending
  marker; the INDEX renderer simply does not fabricate a row with no usable link text or sort
  key (which would render as a zero-width `[]()` anchor and have no stable ordering). This
  keeps SC-006 honest — a row that cannot link or sort is not "reachable," so skipping it (and
  letting the lint surface the real defect) is the reachable-or-flagged outcome, not silent
  half-rendering.
- **FR-016 [US3]**: Only version-marked (gated) SPEC-MOCs MUST be indexed; legacy/
  non-marked SPEC-MOCs MUST be skipped, consistent with the existing gating.
- **FR-017 [US3]**: The generator's `main()` MUST discover and process the roadmap-MOC home
  note in addition to the per-spec MOCs. Discovery is a filename glob
  `docs/ai/specs/*-roadmap-MOC.md` (0..N matches), each gated via the existing
  `structureVersion` gate (`moc_is_gated`), processed **in addition to and disjoint from**
  the existing repo-root `specs/` scan (the two trees do not overlap, so no double-processing
  and a clean no-op when zero home notes exist).
- **FR-017a [US3]**: A home note emitted from the PRSG-002 template ALWAYS carries exactly the
  INDEX sentinel pair present and the PRS + BACKLINKS pairs absent (FR-002), so the normal
  home-note regeneration takes the whole-zone-rewrite path (INDEX present → filled). The
  generator MUST guard the inverse case: a **discovered, gated** home note that does NOT carry
  its INDEX sentinel pair — i.e. all three GENERATED pairs are absent — MUST NOT be sent down
  the inject-if-missing path. That path fires precisely on all-three-absent and would inject
  ALL THREE zones (INDEX + PRS + BACKLINKS) and render PRS/BACKLINKS against the home note's
  non-spec directory (`docs/ai/specs/`), directly contradicting FR-002's "INDEX pair only".
  For a home-note target the generator MUST instead **fail safe (exit 2, no write, actionable
  stderr line naming the offending home note)** when the INDEX pair is absent, because a gated
  home note without its INDEX zone is a malformed home note, not a fresh spec-MOC awaiting
  injection. (An unbalanced INDEX pair — orphan/duplicated sentinel — is already the exit-2
  fail-safe inherited from the shared marker-balance check; this clause additionally covers the
  all-absent case, which the inherited inject-if-missing default would otherwise mishandle for
  a home-note target.)
- **FR-018 [US3]**: The per-spec-MOC code path MUST remain byte-identical to its
  pre-activation output, so the pinned PRSG-003 byte-for-byte contracts stay green; the
  roadmap-MOC INDEX is an additive code path. Because `render_index` already runs against
  every spec-MOC's present INDEX zone, the activation MUST branch on target type: the
  repo-wide INDEX render applies **only** to the home note, and the spec-MOC INDEX zone MUST
  continue to render empty — emitting rows for spec-MOCs would break this guarantee.
- **FR-019 [US3]**: Regenerating an up-to-date home note MUST be idempotent — a second
  run with no intervening edits produces a zero-byte diff.

**Cross-cutting**

- **FR-020 [US1][US2]**: The Codex mirrors MUST convey the SAME content as the Claude Code
  variants, not merely co-exist alongside them. Specifically: the Codex `speckit-prd` mirror
  MUST describe the same home-note emit step (same two-zone model, same curated-zone
  derivation, same epic-cap advisory) as the Claude `speckit-prd`, and the Codex
  `speckit-coach` mirror MUST teach the same two-zone split and the same cap-epics guardrail
  as the Claude `speckit-coach`. Parity is **semantic equivalence**, not byte-identical prose:
  each mirror MAY differ in its own voice and in its runtime's interview mechanism (the Codex
  free-text Q&A loop vs `AskUserQuestion`), but MUST NOT differ in the model, the emit step, or
  the teaching it conveys. (Codex parity is mandatory for the two skills.)
- **FR-021 [US3]**: The generator MUST remain a single shared, runtime-agnostic copy
  referenced by path; it MUST NOT be duplicated into the Codex skills tree, and no new
  separate roadmap-MOC generator script and no shared-library extraction refactor is
  introduced (the existing generator is extended in place).
- **FR-022 [US3]**: The GENERATED INDEX MUST contain link + status only; it MUST NOT render
  a table or dashboard and MUST NOT parse spec H1 titles (the row content is a pure function
  of frontmatter fields, preserving determinism).

### Test Coverage Note (deliberate deviation)

The PR-Size Governance roadmap catalog lists PRSG-004 tests as **L1** (and the design
baseline as L1/L2/L3/L8), both assuming skill-prompt-only changes. Because US3 activates
deterministic generator code (`render_index`), this spec adopts the final test set
**L1, L2, L3, L4, L8** — adding **Layer 4** so that the byte-identical / determinism
guarantees in SC-004 and SC-005 are protected by a deterministic fixture (same committed
SPEC-MOCs → byte-identical INDEX zone; spec-MOC path byte-identical). This is recorded as a
deliberate, adjudicated deviation from the catalog's terse "Tests: L1".

## Assumptions

- **Home note `up:` target** (resolved here): the home note's `up:` points at the
  technical-roadmap (`<slug>-technical-roadmap.md`), and `speckit-prd` adds the reciprocal
  roadmap → home-note link, making the two top-level docs mutually linked. (FR-006.)
- **Generator home-note discovery mechanism** (resolved; confirmed against the generator +
  lib): the extended `main()` discovers the home note by filename glob
  `docs/ai/specs/*-roadmap-MOC.md` (0..N matches), gated via `moc_is_gated` (the
  `structureVersion` bare-int gate — one mechanism, not a separate field), disjoint from the
  repo-root `specs/` scan. `render_index` is **context-scoped**: the home-note INDEX is
  repo-wide over all gated specs (PRSG-004 assumes a single roadmap; per-roadmap scoping for
  coexisting roadmaps is deferred to PRSG-011), while every spec-MOC INDEX stays empty. The
  L4 fixture MUST assert BOTH halves (home-note INDEX fills with repo-wide rows; every
  spec-MOC INDEX stays byte-identical/empty) and that the emitted home note carries only the
  INDEX sentinel pair.
- **coach teaching location** (resolved): the two-zone + cap-epics teaching lives in a
  **new dedicated reference file** `speckit-pro/skills/speckit-coach/references/roadmap-moc-guide.md`,
  NOT inline in `SKILL.md` (keeps the body lean) and NOT appended to an existing reference
  (`command-guide.md` covers only official `/speckit-*` commands; `slicing-heuristics.md` is
  slice-sizing — a different concern, but the precedent for a focused single-topic reference).
  The Codex `speckit-coach` mirror has no own `references/` and links to the shared tree
  (`../../skills/speckit-coach/references/…`), so the file is authored once and Codex parity
  (FR-020) needs no duplicate doc. Discoverability: add a minimal keyword cluster
  ("roadmap map / home note / Map of Content / navigation") to the `description:` of BOTH
  coach `SKILL.md` files (each in its own voice), plus one routing-table row and one
  References-list entry per mirror. A **new Layer-2 trigger case** is added to both
  `evals/speckit-coach-trigger.json` and `codex-evals/speckit-coach-trigger.json` to verify
  the description surface triggers on roadmap-MOC home-note queries.
- **No-phases fallback shape** (resolved): a flat catalog yields a single "Specs" epic with
  an advisory note to group the specs — prd-emission prose only; the generator never touches
  the curated zone. (FR-004.)
- **Codex parity is a constraint, not an open question**: the home-note emission is
  post-interview file writing (runtime-agnostic), so both `speckit-prd` mirrors get the
  identical emit step even though the Codex interview uses a free-text Q&A loop; the
  existing Codex-skills and skill-reviewer gates must stay green.
- **Backfill ownership**: this repo's own PRSG roadmap gets its home note from the later
  retro-migration spec (PRSG-011 Tier-0), not from PRSG-004; the L3 eval demonstrates
  emission on a new fixture roadmap, not on this repo.
- **Reused infrastructure**: the INDEX renderer reuses the existing frontmatter accessor,
  the ID-normalization helper, the sentinel framing, whole-zone regeneration, and the
  atomic-write path already built and tested for the per-spec-MOC zones.
