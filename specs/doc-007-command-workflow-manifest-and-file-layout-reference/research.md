# Research: Command, workflow, manifest, and file-layout reference

## Decision: Generate committed Markdown pages with a docs-site-local Node script

**Rationale**: DOC-007 requires full visible page content, stable reviewable diffs, and deterministic stale-output detection. A single `docs-site/scripts/generate-reference-pages.mjs` script can render Markdown bytes directly, reuse the existing docs-site pnpm environment, and avoid a browser/component-rendered data path.

**Alternatives considered**:

- Hand-authored pages: simpler but does not satisfy generate/check drift detection.
- MDX/component-rendered data: flexible but hides more of the final public prose behind component logic.
- Repo-wide reusable docs platform: too broad for DOC-007 and conflicts with the explicit scope cut.

## Decision: Use an explicit allowlisted source inventory

**Rationale**: The generator must distinguish source evidence from generated output and must avoid `.git`, `.worktrees`, `node_modules`, user-local plugin installs, network inputs, and generated reference pages as evidence. An explicit list of path prefixes and exact files makes the read boundary auditable.

**Alternatives considered**:

- Recursive repo scan: easier to implement initially but risks pulling in fixtures, generated pages, local state, or unrelated docs.
- User-provided JSON inventory: flexible but violates the checked-in-source-only evidence requirement.

## Decision: Separate source facts and inferred notes in the generator data model

**Rationale**: Source facts must be directly supported by checked-in files. Explanatory relationships, such as source-vs-dist responsibility, are useful but must be labeled as inferred notes with `Based on:` paths so generated prose does not overstate evidence.

**Alternatives considered**:

- Single prose field per row: shorter output but blurs evidence with interpretation.
- Wide evidence tables only: compact but less readable for public docs users and less useful for agents deep-linking to sections.

## Decision: Implement read-only check mode by comparing generated bytes in memory

**Rationale**: `reference:check` must fail on stale generated pages without rewriting them. Rendering expected files into memory and byte-comparing against committed output gives deterministic stale detection and keeps the working tree unchanged.

**Alternatives considered**:

- Regenerate then rely on `git diff`: common pattern but mutates files in check mode.
- Build-only validation: proves pages build, but not that generated output is current.

## Decision: Keep validation local to docs-site scripts

**Rationale**: DOC-007 owns local generate/check behavior. DOC-010 owns GitHub Actions and broader docs CI hardening, so the implementation should add `reference:generate`, `reference:check`, and make `validate` run `reference:check` before the existing check/build sequence without editing `.github/workflows/*`.

**Alternatives considered**:

- Add GitHub Actions wiring now: useful later but explicitly out of scope.
- Leave `validate` unchanged: weaker local guard and inconsistent with FR-021.

## Decision: Preserve the existing `/reference/` landing page and sidebar group

**Rationale**: Existing links target `docs-site/src/content/docs/reference.md`. The generated subpages can live under `docs-site/src/content/docs/reference/` while Starlight sidebar order remains explicit: `reference`, generated subpages in stable order, then `glossary`.

**Alternatives considered**:

- Move the landing page to `reference/index.md`: unnecessary route churn.
- Create a separate generated reference group: duplicates the existing Reference IA and risks breaking current mental models.
