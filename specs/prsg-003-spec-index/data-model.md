# Data Model: Generated index/PRs/backlinks + status integration + phase-gate regen

Phase 1 output for PRSG-003. The "data" here is the structure of the generated
zones, their ordering keys, and the one repo-local input entity (the PRS
manifest). All entities are derived from committed repo files — no database.

## Entities

### E1 — Generated zone

A region inside a spec's `SPEC-MOC.md` bounded by a dedicated HTML-comment
`START`/`END` pair. Three kinds exist; each is independently positioned and
**wholly** replaced on each rebuild (FR-001/FR-002).

| Field | Value |
|-------|-------|
| Kind | one of `INDEX`, `PRS`, `BACKLINKS` |
| Start marker | exact line, e.g. `<!-- GENERATED:INDEX:START (do not edit; regenerated by generate-spec-index.sh) -->` |
| End marker | exact line, e.g. `<!-- GENERATED:INDEX:END -->` |
| Body | rendered content (may be empty/link-free); fully replaced each run |
| Presence | a missing marker pair → that zone is skipped (FR-009); never half-written |

Fixed zone order in a map note: **INDEX → PRS → BACKLINKS** (FR-008/FR-017, D2).
Empty zone = the two marker lines on consecutive lines with no body between them
(link-free; keeps the stale-index lint green).

### E2 — Spec map note (`SPEC-MOC.md`)

The per-spec map-of-content note that hosts the generated zones. **In scope** iff
it is version-marked (`structureVersion` bare-integer ≥ 1, per
`moc_is_gated`). The generator may inject the three empty zones into an in-scope
note that lacks them (FR-008), at the canonical anchor, exactly once, idempotently.

| Field | Source | Notes |
|-------|--------|-------|
| `up:` | frontmatter | NOT touched by the generator (orphan lint owns it) |
| `spec_id` | frontmatter | used only to confirm in-scope identity; matches dir (PRSG-002) |
| `structureVersion` | frontmatter | gate: ≥ 1 → in scope; absent/0/malformed → skipped + unmodified (FR-007) |
| body | file | the generator appends/refreshes the three-zone block at the end |

### E3 — Reachability entry (BACKLINKS row)

A relative `[]()` link from a spec's map note to one of that spec's own artifacts.
Rendered relative to the map note's own directory so it resolves under
`specs/<branch>/` (keeps the stale-index lint green).

| Field | Value |
|-------|-------|
| Link text | the artifact's display label (e.g. `spec.md`, `research`, `prs manifest`) |
| Link target | path relative to the MOC dir (e.g. `spec.md`, `research.md`, `.process/prs.json`) |
| Precedence bucket | spec / plan / tasks / data-model / research / contracts / checklists / process |
| Existence | only files that exist on disk are enumerated (no dangling links) |

Per-spec ordering (FR-005, D2/R5): fixed artifact precedence —
`spec.md` → `plan.md` → `tasks.md` → `data-model.*` → `research.*` →
`contracts/**` → `checklists/**` → `.process/**` — then lexicographic path within
each bucket. Independent of filesystem enumeration order (SC-009): file lists are
`LC_ALL=C sort`-ed before rendering.

### E4 — Pull-requests record (PRS row)

A repo-local committed datum mapping a slice → PR number → merged commit, read
from `specs/<branch>/.process/prs.json`. PRSG-003 *renders* it; *who writes* it is
PRSG-009 (out of scope here).

| Field | Type | Notes |
|-------|------|-------|
| `slice` | string | a spec/slice identifier; normalized via `moc_normalize` for ordering |
| `pr` | integer | PR number |
| `merged_sha` | string | short or full merged commit SHA |

Rendered as **plain text** (not a `[]()` link), e.g. `PRSG-003 · PR#117 · abc1234`.
Ordering: by normalized `slice` ascending, then `pr` ascending. Absent file or
`records: []` → empty-but-valid (link-free) PRS zone (FR-011).

### E5 — Normalized identifier (sort/match key)

The canonical `(namespace, number-suffix)` key produced by `moc_normalize`
(PRSG-002), used to order cross-spec lists (INDEX) and PRS rows, and to match
specs across the repository. **Reused, never reimplemented** (FR-004).

| Property | Value |
|----------|-------|
| Producer | `moc_normalize` from `tests/lib/moc-id-normalize.sh` |
| Form | `"<namespace> <number-suffix>"` (space-separated) |
| Use | ascending sort key for INDEX entries and PRS `slice` ordering; equality match for joins |

## Discovery & scope rules

- **Discovery (FR-007):** enumerate spec directories under `specs/`; a directory
  is in scope iff its `SPEC-MOC.md` is version-marked (`moc_is_gated`). Non-marked
  / legacy specs are skipped and left unmodified (no injection, no zone write).
- **Enumeration boundary (FR-018):** a spec's BACKLINKS enumerates ONLY that
  spec's own `specs/<branch>/**` tree (including `.process/`). Never another tree.
- **INDEX dormancy (FR-019):** the roadmap-level INDEX path is built and
  fixture-exercised but renders nothing live until a roadmap home note carrying
  the INDEX markers exists (PRSG-004). In a spec-MOC the INDEX zone is present but
  renders empty/link-free here.

## State transitions (a single generator run over one map note)

```text
read map note
  ├─ not a regular readable file ─────────────→ ERROR (exit 2), no write (FR-016)
  ├─ not version-marked ──────────────────────→ SKIP, unmodified (FR-007)
  └─ version-marked:
       ├─ missing all/any zone markers ───────→ inject missing zones at anchor (FR-008),
       │                                          present zones still rebuilt (FR-009)
       └─ render each present zone body:
            INDEX     → dormant/empty here (FR-019)
            PRS       → from .process/prs.json, or empty-but-valid (FR-010/FR-011)
            BACKLINKS → sorted reachability over specs/<branch>/** (FR-005/FR-006/FR-018)
       → whole-zone splice (FR-002) → compare to committed:
            --check mode → diff only; report current(0)/stale(1); write nothing (FR-012)
            write mode   → write if changed; re-run yields zero diff (FR-003/SC-001)
```
