# Error-Handling Checklist: Generated index/PRs/backlinks + status integration + phase-gate regen

**Purpose**: Unit-tests-for-requirements covering error-handling concerns specific to
this feature — the 3-way result enum (current / stale / error) and its
never-conflate discipline, the errtrace/ERR-trap mechanism that keeps an internal
failure out of the stale signal, missing vs unbalanced sentinel pairs, malformed /
unreadable / non-regular-file inputs, the `--check` no-write-on-error invariant,
jq/invalid-JSON vs absent/empty PRS data, write atomicity (no partial write), and
the empty-tree no-op. Each item interrogates whether the **requirements** (in
spec.md / plan.md) are complete, clear, consistent, and measurable — NOT whether
the implementation works.
**Created**: 2026-06-06
**Feature**: [spec.md](../spec.md) · [plan.md](../plan.md)

**Note**: This checklist is generated by the `/speckit-checklist` command based on
feature context and requirements. Items reference spec sections `[Spec §X]` /
`[Plan §X]` where a requirement exists, or carry a `[Gap]` marker where a
requirement is missing or under-specified on the evaluated surface (spec.md /
plan.md). Concerns already covered by the data-integrity checklist (malformed-PRS
Edge Case, SC-011 lint-cleanliness, `LC_ALL=C sort` / opaque-suffix / schemaVersion)
are not duplicated here.

## 3-Way Result Enum (current / stale / error — never conflated)

- [ ] CHK001 Is the three-outcome result enum (current, stale, error) specified as a hard requirement the generator MUST expose distinguishably? [Completeness, Spec §FR-015]
- [ ] CHK002 Is each outcome bound to a concrete, distinguishable exit code (0 current/clean, 1 stale, 2 error) in a requirement-visible place the spec or plan points to? [Clarity, Plan §contracts/generator-cli.md]
- [ ] CHK003 Is the never-conflate invariant — exit 2 (error) is NEVER reported as exit 1 (stale) and vice-versa — stated as an explicit requirement, not only implied by the table? [Consistency, Spec §FR-021, Plan §contracts/generator-cli.md]
- [ ] CHK004 Is it specified that an unexpected internal/operational failure surfaces as the ERROR result (exit 2) and never as the benign content-difference STALE result (exit 1)? [Clarity, Spec §FR-021]
- [ ] CHK005 Is the stream discipline specified (error → actionable message on stderr; stale → which-maps/which-zones on stdout) so the two outcomes are not conflated on the output channel either? [Completeness, Plan §contracts/generator-cli.md]
- [ ] CHK006 Are the per-consumer actions for each of the three outcomes specified for `speckit-status` (current → report current; stale → "run regen"; error → surface, write nothing)? [Coverage, Spec §FR-013, Plan §contracts/generator-cli.md]
- [ ] CHK007 Are the per-consumer actions for each of the three outcomes specified for the `speckit-autopilot` phase gate (current/diff-empty → no commit; non-empty → fold into checkpoint; error → surface and stop, do NOT commit a broken regen)? [Coverage, Spec §FR-014, Plan §contracts/generator-cli.md]
- [ ] CHK008 Is the requirement to distinguish a *non-empty rebuild diff* (a write-mode "stale-equivalent" that triggers a commit) from an *error* stated for the write path, so a failed regen is never committed as if it were a normal diff? [Consistency, Spec §FR-014/FR-021]

## Internal-Failure Trap Discipline (errtrace / ERR trap → exit 2)

- [ ] CHK009 Is there a requirement that an unexpected `set -e` failure inside the generator is mapped to the error result (exit 2) rather than leaking an arbitrary exit code? [Completeness, Spec §FR-021, Plan §D5]
- [ ] CHK010 Is the `set -E` (errtrace) + ERR-trap mechanism — propagating the trap into shell functions so an internal failure trips it — pinned as the required implementation discipline in the plan? [Clarity, Plan §D5 / §contracts/generator-cli.md]
- [ ] CHK011 Is the PRSG-002 lint precedent (`validate-moc-orphan.sh`'s `set -E` + `_on_err` → exit 2 on stderr) named as the convention this generator MUST match, so the two failure classes stay unconflated the same way? [Traceability, Plan §D5]
- [ ] CHK012 Is it specified that the trap is disarmed before any deliberate non-zero exit (the stale exit 1) so a legitimate stale result is never remapped to error exit 2? [Clarity, Plan §D5]

## Missing vs Unbalanced Sentinel Pairs

- [ ] CHK013 Is the missing-marker-pair behavior specified — a map note missing an individual zone's pair has that one zone skipped while other present zones are still rebuilt? [Completeness, Spec §FR-009]
- [ ] CHK014 Is the no-marker / no-version-marker map note specified as skipped silently (exempt), distinct from an error? [Clarity, Spec §FR-007]
- [ ] CHK015 Is the behavior for an UNBALANCED / partial sentinel pair (a `START` without its matching `END`, or vice-versa) defined as a safe, non-corrupting outcome rather than left undefined? [Completeness, Spec §FR-022 / §Edge Cases]
- [ ] CHK016 Is the behavior for a DUPLICATED or NESTED sentinel pair (e.g. two `GENERATED:PRS:START` lines, or an `END` before its `START`) defined so the splice cannot mis-bound a zone or corrupt the file? [Coverage, Spec §FR-022]
- [ ] CHK017 Is the result class for an unbalanced/duplicated/nested pair stated unambiguously and consistently — the malformed-target fail-safe error (FR-016), explicitly distinguished from FR-009's *missing* (absent) pair which only skips that zone? [Consistency, Spec §FR-022 / §FR-009]
- [ ] CHK018 Is full-line string-equality sentinel matching (not a loose substring/regex) stated as the guard that prevents a `GENERATED:` token in prose from being mistaken for a real sentinel boundary? [Clarity, Plan §D1]

## Malformed / Unreadable / Non-Regular-File Inputs

- [ ] CHK019 Is the malformed/unreadable source map note (target) failure specified to fail safe with an actionable message and a non-success (exit 2) result? [Completeness, Spec §FR-016]
- [ ] CHK020 Is the non-regular-file target case (a directory or symlink where a map note is expected) specified as rejected rather than written through? [Coverage, Spec §FR-016 / §Edge Cases]
- [ ] CHK021 Is the malformed/unreadable PRS manifest failure specified to fail safe with an actionable message and exit 2, distinct from the absent/empty case? [Completeness, Spec §Edge Cases / Plan §contracts/prs-manifest.schema.md]
- [ ] CHK022 Is "actionable message" given enough definition (names the offending file/path and the failure class) to be objectively satisfied, rather than left as a vague adjective? [Clarity, Spec §FR-016]
- [ ] CHK023 Is each malformed-input failure required to NOT partially write or corrupt the target, consistently across all three input classes (map note, PRS manifest, non-regular-file target)? [Consistency, Spec §FR-016]
- [ ] CHK024 Is an unreadable PRS manifest that *exists* but cannot be read treated under the malformed/error path (exit 2), and not silently collapsed into the absent → empty-but-valid path? [Coverage, Plan §contracts/prs-manifest.schema.md]

## `--check` Never Writes on Any Error Path

- [ ] CHK027 Is the `speckit-status` read-only contract (writes zero files in current, stale, AND error cases) stated as a measurable outcome the status consumer must preserve? [Measurability, Spec §SC-003/SC-004]

## Write Atomicity (no half-written / corrupted map note)

- [ ] CHK033 Is a no-partial-write / atomic-update requirement stated for the write path, so a write that fails partway cannot leave a half-written or corrupted map note? [Completeness, Spec §FR-016]
- [ ] CHK034 Is the atomicity MECHANISM (write-to-temp-then-rename, or an explicitly-named equivalent) pinned in the plan? [Clarity, Plan §D6]
- [ ] CHK035 Is the half-updated zone called out as the exact failure mode being eliminated, tying the atomic-write requirement back to the spec's "partial in-place edits are forbidden" rationale? [Traceability, Spec §FR-016 / §US1 / §FR-002]
- [ ] CHK036 Is the atomic-write guarantee specified to hold per target across a multi-map run, so a failure writing one spec's map cannot leave an earlier or later spec's map half-written? [Coverage, Spec §FR-016, Plan §D6]
